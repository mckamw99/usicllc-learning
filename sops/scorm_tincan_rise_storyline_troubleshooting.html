Completion Tracking for Rise & Storyline

Why SCORM and Tin Can (xAPI) packages built in Articulate Rise and Storyline get stuck in Docebo — and how to publish them so completion reports cleanly every time. The fix is almost always made at publish time, not in the LMS.

The Handshake: Who Decides “Complete”

Section 1

A SCORM or xAPI package is a self-contained program that reports its own status to the LMS. Docebo only displays what the package tells it. The rule that decides completion — view 90% of slides, pass a quiz, reach a trigger — is baked into the package when you publish from Rise or Storyline. So when a course won’t flip to Completed, the cause is usually one of three things, in this order of likelihood:

1. The package never sent its status

The learner closed the tab instead of pressing Exit / Close / Submit, a session timed out, or connectivity/cookies/Safari blocked the call. The completion exists on screen but never reached Docebo.

2. The package’s internal completion rule wasn’t actually met

Tracking is set to “slides viewed” but a free-navigation menu let the learner skip the threshold; or it’s set to “quiz passed” and they didn’t pass; or a completion trigger sits on the wrong slide.

3. The reported status doesn’t map to what Docebo expects

The wrong reporting option or SCORM version is sending a status (e.g. passed vs completed) that the platform reads differently than intended — the mastery-score trap below is the classic example.

Golden rule: test every package in SCORM Cloud first (it’s the industry-standard reference engine), then in a Docebo sandbox course, before publishing to learners. If it reports correctly in SCORM Cloud but not Docebo, the issue is platform-side; if it fails in both, it’s the package.

Pick the Right Standard for USIC Academy

Section 2

Both Rise and Storyline can publish to AICC, SCORM 1.2, SCORM 2004, xAPI (Tin Can), and cmi5. For USIC Academy the practical choices are SCORM 1.2, SCORM 2004 3rd Edition, or xAPI.

StandardStrengthsWeaknessesUSIC fit
SCORM 1.2Lightweight, low bandwidth, widely compatible.Single status value (LMS must interpret); limited suspend/bookmark data, so long courses lose progress; inconsistent completion vs success.Short courses only
SCORM 2004 3rd Ed.Separate completion & success statuses; large suspend-data limit for reliable bookmarking on long packages.More bandwidth; still subject to constant-communication fragility.Default for SCORM
xAPI (Tin Can)Most robust completion/bookmarking; Docebo handles it natively (no separate LRS needed), keeping calls separate per user/course.Multi-module packages need the right Docebo tracking toggle (see Section 8).Best for reliability
SCORM 2004 edition matters at USIC. USIC Academy supports SCORM 1.2 and SCORM 2004 3rd Edition. Although Articulate generally recommends 2004 4th Edition, publish USIC SCORM 2004 content as 3rd Edition — 4th Edition has known stability issues on the platform. Avoid SCORM 1.2 for any course longer than a few minutes because of its bookmarking limits.

Reporting Options — Completion vs Success

Section 3

In Rise (the Reporting dropdown) and Storyline (the Reporting and Tracking button) you choose one of four status pairs. Completion status answers “did they finish?”; success status answers “did they pass?”

Reporting optionBehaviorUse when…
Passed / Incomplete RecommendedReports Passed on success, Incomplete until then. Most likely to record both statuses cleanly.Completion-based courses
Passed / Failed RecommendedReports a pass/fail outcome. May show Unknown success until the learner finishes, then resolves.Graded / quiz courses
Completed / IncompleteReports Completed/Incomplete only — no pass/fail. Beware the mastery-score trap on SCORM 1.2 with a quiz.Ungraded completion
Completed / FailedLess common; mixes completion with a fail outcome.Rare
Why 1.2 is inconsistent: SCORM 1.2 carries a single status, so the LMS has to guess whether a value means “completed” or “passed.” SCORM 2004 carries completion and success separately, removing the guesswork — another reason to prefer 2004 3rd Edition or xAPI for anything graded.

The Mastery-Score Trap (SCORM 1.2 + Quiz)

Section 4

This one catches everyone. When you publish a Rise or Storyline lesson that contains a quiz as SCORM 1.2, Articulate automatically inserts a mastery score into the package manifest. The SCORM spec then requires the LMS to override the status with pass/fail — so the course will never display “Completed,” even if you set the reporting option to Completed/Incomplete.

Symptom: a learner passes the quiz, but Docebo shows the course as still In Progress or only ever as Passed (never Completed), and your “Completed” reporting filter misses them.

Fix

Publish the quiz course as SCORM 2004 3rd Edition (separate completion & success means the mastery score no longer forces an override of completion), and report as Passed/Failed. If you must stay on SCORM 1.2, treat Passed — not Completed — as your indicator of success, and set the Docebo course’s expectations and reports accordingly.

Storyline: Reporting & Tracking Settings

Section 5

In Storyline’s publish dialog, the Reporting and Tracking button is where completion is decided. You can track by:

Tracking methodWhat it means & the catch
Number of slides viewed (%)Completes when the learner views N slides. Catch if the menu allows free navigation, learners can skip to the end and either never hit the threshold or hit it without engaging.
A quiz resultCompletes based on a Results slide score. The most reliable for graded content — sends real score/interaction data.
A completion trigger (“Using triggers”)Completes when a Complete course trigger fires. Catch a single trigger sends no quiz/interaction data, and a recent SL360 behavior can cause it to fail to report if it’s on the wrong slide.
Multiple tracking methodsSL360 can combine methods (e.g. complete when quiz passed or trigger fired). Useful but test carefully.
If a “Complete course” trigger isn’t reporting: put the trigger on a Results slide, add a Submit Interaction trigger to that slide, and switch tracking to “When the learner completes a quiz” rather than “Using triggers.” This restores reliable reporting and sends interaction data.
Don’t complete on timeline starts. A “complete course when timeline starts on slide 1” trigger marks the course complete the instant it opens — learners get credit for doing nothing. Trigger completion at a genuine end point (Results slide / final slide timeline ends).

Rise: Export & Tracking Settings

Section 6

In Rise, Publish › LMS lets you choose the standard and a Tracking option. All content can be tracked by completion percentage or Storyline block; courses can also be tracked by quiz result.

Rise makes you choose one or the other. A single Rise package can track either completion percentage or quiz result — not both. The LMS reads the whole Rise course as one object, so you can’t capture per-lesson quiz scores from a single SCORM package. If you need a graded score, track by quiz result; if you only need a finish, track by completion percentage.

Three options in Rise’s More settings menu solve less-common stuck states:

Reset Learner Progress

If you delete a lesson and re-upload the course, some learners may see a blank page. Toggle this on so their progress resets cleanly on next launch (quiz data is retained).

Hide LMS Interface

SCORM 2004 exports can show duplicate Suspend/Continue/Close buttons. Toggle this to hide the extras.

Only Load in LMS

Allows extra windows outside the training — but note that quiz and completion tracking won’t work in those additional windows, which can look like a stuck completion.

Rise SCORM 1.2 packages with Restrict Navigation have been reported to occasionally skip lessons and lose progress on long courses. Prefer SCORM 2004 3rd Edition or xAPI for multi-lesson Rise courses.

The Docebo-Side Settings That Interact

Section 8

xAPI “Track all activities” toggle

On the xAPI training material’s settings (Course management › course › Training material › the xAPI object › Settings). On (default): completing the first activity completes the whole package — right for a single Rise/Storyline export with one completion statement. Off: every sub-activity must be completed. Match it to how the package was authored.

End Object Marker

Set the Rise/Storyline package as the course’s end object marker so completing it marks the whole Docebo course complete, independent of any optional materials. Not retroactive.

Reset tracking / Manual completion

For a clean redo, reset the material in Reports › User statistics (watch Central Repository sharing). For a confirmed reporting break on sound content, manually set the enrollment to Completed, or bulk-remediate via the bulk_complete.py toolkit pattern with an audit file.

Remember the SCORM learner instruction: the package only records when the learner presses its own Close / Exit / Submit button. Build that button into the final slide/lesson and call it out in enrollment communications.

Pre-Publish QC Checklist

Section 10
  1. Confirm the standard & editionSCORM 2004 3rd Edition (not 4th) for USIC, or xAPI. Avoid SCORM 1.2 for anything beyond a few minutes.
  2. Pick a deliberate tracking methodQuiz result for graded; completion % / slide count for informational. Make sure the completion point is a real end of the content.
  3. Set reporting to Passed/Incomplete or Passed/FailedAnd remember the SCORM 1.2 + quiz mastery-score trap if you ever must use 1.2.
  4. Include a clear Exit/Submit stepBuild the final action that fires completion, and tell learners to use it.
  5. Test in SCORM CloudVerify the status resolves to the expected completion/success before touching Docebo.
  6. Test in a Docebo sandbox courseConfirm the Track-all-activities toggle (xAPI), end object marker, and that completion lands — on desktop Chrome.
  7. Freeze the package once liveDon’t edit a published SCO mid-rollout; schedule updates and reset progress in a maintenance window.
