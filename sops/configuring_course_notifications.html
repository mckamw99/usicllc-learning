Configuring Course Notifications

How to create, target, schedule, and verify automated notifications for a course in Docebo — so the right learners and managers hear about enrollments, deadlines, completions, and sessions at the right time.

Overview

Section 1

A notification is an automated message that Docebo sends when a specific event happens in the platform — for example, a learner is enrolled in a course, a deadline is approaching, or a course is completed. Notifications keep learners on schedule and keep instructors and managers informed without anyone watching the platform manually.

Each notification is tied to a single event and is delivered through one or more channels (in-platform, email, mobile push, or Slack). Notifications are managed centrally from the Notifications area — you do not build them from inside an individual course. Instead, you create one notification and then use trigger filters to scope it to the specific course(s) you care about.

This guide reflects Docebo’s current notification service. If your screens differ, you may still be on the legacy notification experience — raise a ticket with the LAT team (learningalerts@usicllc.com) before proceeding.

Before You Start

Section 2

Prerequisites

  • 1
    The Notifications app is activeThe Notifications app must be turned on for the platform (managed under Add-ons and integrations › Docebo additional features). On USIC Academy this is already active — no action needed.
  • 2
    You have the right permissionsYou need to be a Superadmin, or a Power User whose profile includes the notification permissions. At USIC, Content Creators can create and edit notifications for their assigned resources.
  • 3
    The *.docebosuite.com domains are reachableThe notification management page will not load if your network blocks *.docebosuite.com. If the page is blank, contact IT to add these domains to the allow list.
Power Users see a filtered list. As a Power User you only see and manage notifications whose trigger resources (courses, branches, groups) are also assigned to you. If a notification you expect is missing, it is almost always a resource-assignment gap — see Scope & Power Users.

Anatomy of a Notification

Section 3

Every notification is built from six parts. Knowing these up front makes the build steps fast.

Event type

The platform event that triggers the message — e.g. User enrolled in a course. Chosen at creation and read-only afterward.

Recipients

Who receives it, defined by relationship to the event: Users, Power Users, Managers, Instructors, or Superadmins. One role per notification.

Delivery channel

How it’s sent: in-platform (bell icon), email, mobile push, or Slack. At least one is required to activate.

Message

The subject and body, personalized with shortcodes such as {{first_name}} and {{course_name}}.

Timing

When it goes out: at the time of the event, a set period before/after, manually, or on a digest schedule.

Trigger filters

Which courses and which user groups/branches the notification applies to. This is how you scope it to one course instead of all of them.

Create a Course Notification

Section 5

Worked example: send an email and in-platform message to a learner the moment they are enrolled in a specific course.

  1. 1
    Open the Notifications areaFrom the Navigation menu, go to Configuration (spanner icon) › Automation › Notifications. The Notifications tab lists every notification you can see, with its event, channels, and status.
  2. 2
    Start a new notificationClick the + icon in the top-right corner. In the Create notification panel, choose the Event category (Courses) and then the Event type (e.g. User enrolled in a course).
  3. 3
    Name it distinctivelyThe Name pre-fills with the event type, but you’ll likely have several notifications per event. Use a clear name such as Enrolled — Decisive Driving (Email + In-app). Click Create and edit. Heads up the event is now read-only — to change it you must create a new notification.
  4. 4
    Set the recipientsOn the Recipients tab, choose the audience — for an enrollment confirmation, select Users (the enrolled learner). You can only choose one role per notification; to also notify Managers, duplicate this notification and set Managers there.
  5. 5
    Compose the message and pick channelsGo to Properties › Composition. Tick Enable email as delivery channel, set the Sender name and a sender address already configured in Domain management, then write the Subject and Body. Repeat for Platform notification if you want the in-app bell message too. Insert shortcodes (next section) to personalize.
  6. 6
    Set the timingOpen Properties › Details and timing and use the Scheduler. For an enrollment confirmation, choose At the time of the event. Other events let you send Before or After by a set number of hours, days, weeks, or months — with a send time and time zone.
  7. 7
    Scope it with trigger filtersOn the Triggers and filters tab, click + and add a Courses and learning plans filter, selecting only the specific course. Optionally add a Users filter to limit it to certain branches or groups. Without a content filter this notification will never fire; with the wrong filter it fires for every course.
  8. 8
    ActivateUse the status toggle at the bottom-right to switch from Inactive to Active. Only active notifications send, and a notification needs at least one delivery channel before it can be activated.
Reuse instead of rebuild. From the notification list, use the ellipsis › Duplicate to clone an existing notification. You choose which parts to copy (timing, filters, recipients, channels) — ideal for sending the same message to a different audience or a different course.

Personalizing with Shortcodes

Section 6

Shortcodes are double-brace tags such as {{course_name}} that are swapped for live data when the message sends. The set available depends on the event type. Common course-related ones:

ShortcodeInserts
{{first_name}} / {{last_name}}The recipient’s first / last name.
{{username}}The recipient’s username (at USIC, the employeeID@usicllc.com format).
{{course_name}}The title of the course that triggered the notification.
{{course_link}}A link to the course. See the link caveat below.
{{event_date}}(ILT) Name, date, start/end time, and location of a single event — in both the learner’s and the event’s time zone.
{{session_dates}}(ILT) The full set of event dates for the session.
{{expire_at}}An expiration date/time, shown in the recipient’s time zone.
Link shortcodes insert plain text, not a clickable link. Most email clients auto-link a bare URL, but to guarantee a clickable link, use the toolbar Insert/edit link button and paste the shortcode in the URL field, or use Source code and wrap it: <a href="{{course_link}}">Open the course</a>. Note that a course link sends the user to an error if they aren’t enrolled — it is not a self-enroll deep link.

Scope & Power User Rules

Section 7

Trigger filters are the single most important — and most error-prone — part of a course notification. Get them wrong and you either blast every course or send nothing at all.

Scoping logic

No content filter = never sends. A notification with no course or learning-plan filter targets no content and will never fire, even when Active.

Multiple branches/groups use OR logic. If you list several user branches or groups, a learner only needs to belong to one of them to qualify.

“All courses” / “All users” is Superadmin-only. Power Users cannot set the All option — they must pick from a custom selection of their assigned resources.

For USIC Content Creators (Power Users): you only see a notification if every resource set as its trigger filter is also assigned to you. The Power Users recipient role can only be set by a Superadmin — you cannot select or change it. If you need an all-courses notification or a Power-User-targeted one, route it to the LAT team.

Best Practices

Section 8
Do
  • Deactivate a notification before editing it — editing recreates the scheduled job.
  • Give every notification a distinctive name so it’s findable in the list and the logs.
  • Test on yourself first: enroll a test account and confirm the message, links, and shortcodes render correctly.
  • Scope to the exact course (and branch/group) every time before activating.
  • Use a sender address already verified in Domain management.
Don’t
  • Don’t leave the content filter empty — it will silently send to no one.
  • Don’t try to target multiple recipient roles in one notification — duplicate it instead.
  • Don’t exceed the 65,535-character limit (HTML and expanded shortcodes count toward it).
  • Don’t rely on a bare link shortcode for a guaranteed clickable link.
  • Don’t change a notification’s event after creation — it’s locked; build a new one.
Bulk enrollment & backfills (USIC). Before a large enrollment or completion backfill — including script-driven runs such as bulk_complete.py — temporarily deactivate the relevant enrollment/completion notifications so you don’t flood hundreds of learners or their managers. Re-activate them once the run is finished. This is also the standard approach when enrolling users into not-yet-published courses to preserve enrollment history.

Verify & Troubleshoot

Section 9

After a notification goes live, confirm it’s actually being delivered using the per-notification Logs tab.

  1. 1
    Open the logsFrom Configuration › Automation › Notifications, click the notification name, then the Logs tab. You’ll see recipient, status, sent date, message ID, and any error detail.
  2. 2
    Read the statusQueued awaiting dispatch   Delivered handed off to the mail server   Failed not sent — check the More information column for the reason.
  3. 3
    Filter and exportFilter by Failed to isolate problems. A Superadmin can export the log (date range up to 14 days, within the last 3 months) for auditing.
SymptomLikely cause & fix
Nothing is sendingNotification is Inactive, has no delivery channel, or has no content/user trigger filter. Confirm all three.
Notification page is blank*.docebosuite.com is blocked on the network. Ask IT to allow-list it.
A Power User can’t see a notificationThe trigger resources aren’t all assigned to that Power User. Align resource assignments.
Status shows FailedBad recipient address, or custom SMTP timeout / auth issue. Review the More information column.
It fired for the wrong coursesContent filter too broad. Edit the Triggers and filters tab and narrow the selection.
In-platform (bell) notifications are retained for about 90 days, after which they’re removed automatically. A Delivered status confirms hand-off to the mail service only — not final inbox delivery.
